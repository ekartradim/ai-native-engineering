Context & session management
critical
Context rot — open new sessions per feature
As the context window fills, models lose track of earlier instructions and start making more mistakes. This is called context rot. Always start a fresh session for each new feature or significant task. Never let a single session grow unbounded.
Carry context forward with @ previous chats
When you need context from a previous session, use @ find previous chats. The agent will summarise the relevant parts and inject them into the new session — no copy-pasting, no re-explaining from scratch.
Limit active MCPs — tool schemas eat context
Every enabled MCP server's tool descriptions are injected into the context window at session start, regardless of whether you use them. Keep only the MCPs you actually need for the current session active. Disable the rest.
critical
Always give the agent up-to-date docs
For any specific technology, use @ docs to add live documentation from the internet. Model training data goes stale fast — up-to-date context is the key to accurate, non-hallucinated output. You can add any URL as a doc source.
Model selection & cost
Smartest models for heavy thinking
Use Opus 4.6 for planning complex features, deep codebase analysis, and security audits. The investment pays off when precision matters. Reserve it for the work where mistakes are expensive.
Cheaper models for execution
Once the plan is ready, switch to Sonnet or Haiku for implementation tasks. Plan with the best, build with the pragmatic. For simple questions use Gemini Flash — very cheap, fast enough.

critical
Use AI-native technologies
If you don't know the technology, pick the AI-native one. Python, React, TypeScript are trained on far more code than niche stacks. Agents are best in these — fewer hallucinations, better tooling, faster iteration.
Agent mindset
critical
Think of the agent like a human collaborator
Give the agent the same tools and context you have so it can do the job properly. A human joining cold can't work without access to the codebase, docs, and a clear brief. Neither can the agent.

critical
Be maximally specific — models cannot read your mind
When asking for changes without a proper plan, vague instructions produce vague (or wrong) results. Specificity is kindness to the model. State what you want, what you don't want, what constraints apply, and what success looks like.
AI-first mindset — use skill creator, not custom code
You don't need to hand-write agents, hooks, or skills from scratch. Use the skill creator to build them. The same applies to subagents and hooks. Let the AI build your AI tooling — that's the compound advantage.
You can revert changes during a session
In agent mode, changes during a session can be rolled back easily before committing. Use this safety net — experiment, inspect the diff, revert what you don't want. You don't lose work between session checkpoints.
Project hygiene & memory
critical
Commit after every workflow step with git
Commit after every completed step: plan → work → test → code review → docs + compound. Granular commits mean easy rollbacks and recoverable code that was accidentally rewritten.
critical
Keep project structure clean after architectural changes
Mess compounds just like knowledge does — but in the wrong direction. Clean the structure after every bigger architectural change. A degraded project layout makes future sessions slower and more error-prone.
Maintain documentation.md, features.md, project-structure.md
Update these after each feature. New sessions can read these three files instead of scanning the whole codebase. Add hooks that auto-read them on session start, or auto-update them on commit. Reduces context cost, improves accuracy.
Build a compounding system
Every session should leave the system better than it started: improve skills, refine hooks, tighten rules, build project-specific agents, update memory. The compound effect is the moat.
Security & quality
critical
Security & code quality audits
Run regular security and code quality audits. Build specialised commands, hooks, or agents for this work so it runs consistently — not ad-hoc. Proper security guardrails should be a permanent part of the harness.
Be careful with dependencies
Supply chain attacks are common. Review what you install. Prefer well-maintained packages with clear ownership. Don't let the agent install arbitrary dependencies without review — treat package.json and requirements.txt as attack surfaces.
Visualise architecture with Mermaid
Use mermaid diagrams to render parts of your solution architecture. Easier to spot inconsistencies visually than in code. Helps both you and the agent understand how components relate before making changes.
Tooling & developer experience
critical
termcolor for console logging
Use termcolor for all console output. Coloured logs make errors immediately visible to both you and the agent. Debugging is faster when red means error and green means pass — without scanning walls of monochrome text.
Browser MCP for frontend work
For frontend development, always enable the built-in browser MCP. The agent can see exactly what you see — including network logs, console errors, and DOM state. Closes the feedback loop that otherwise requires constant manual copy-pasting.
Compound knowledge — save lessons learned
After every task, save reusable lessons to structured .md files in a dedicated folder. Each run leaves behind institutional knowledge. The system gets faster and smarter with every session — but only if you capture it deliberately.
