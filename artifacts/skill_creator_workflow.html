Skill Creator

11 phases
looping eval cycle
subagent-driven
Phase 1–3 · understand & draft
01
Capture intent
entry point
Four questions answered first
What should this skill enable Claude to do?
When should it trigger? → drives description
What is the expected output format?
Should test cases be set up?
Smart entry detection
Detects where in the loop the user already is
Extracts steps from conversation history if workflow already happened
Evals recommended for objective outputs; skipped for subjective (style, design)
02
Interview & research
gather
Questions asked
Edge cases and failure modes
Input / output formats and example files
Success criteria and dependencies
Parallel research (if MCPs available)
subagents invoked in parallel for docs + similar skills
Test prompts not written until interview complete
03
Write SKILL.md
draft
Frontmatter fields
name kebab-case, max 64 chars
description primary trigger — write assertively, combat undertriggering
compatibility required tools (optional, rare)
Directory structure
SKILL.md body <500 lines; load on trigger
scripts/ executable code; deterministic tasks
references/ docs loaded into context on demand
assets/ templates, fonts, icons used in output
Phase 4–8 · evaluation run · one continuous sequence
04
Write test cases
evals
Process
2–3 realistic user prompts drafted and reviewed with user
Saved to evals/evals.json
Assertions left EMPTY at this stage — written during runs
Query design rules
Realistic, not sanitised — write how a real user would phrase it
Complex enough that a skill would genuinely help
05
Run parallel evaluations
execute
Parallel subagent pairs per test case
with_skill receives SKILL.md + eval prompt + input files
without_skill baseline — no skill provided (new skill) or
old_skill baseline — prior version snapshot (iteration)
Both spawned in same turn so they finish simultaneously
Workspace layout
iteration-N/eval-ID/with_skill/
iteration-N/eval-ID/without_skill/
eval_metadata.json written per eval dir
06
Draft assertions (while runs in progress)
concurrent
Good assertions are
Objectively verifiable — no human judgment required
Descriptively named — readable as-is in viewer
Scripted where possible (faster, reusable, reliable)
Skip assertions for
Subjective output skills — writing style, design quality
Results saved back into evals/evals.json before runs finish
07
Capture timing data
ephemeral
On subagent completion
Notification arrives with total_tokens and duration_ms
Written immediately to timing.json in run dir
Not persisted elsewhere — cannot be recovered after the fact
08
Grade → aggregate → analyse → launch viewer
pipeline
Four sequential steps (no stopping)
8a Grade — grader subagent reads agents/grader.md; writes grading.json; fields: text, passed, evidence
8b Aggregate — aggregate_benchmark.py → benchmark.json + benchmark.md; pass_rate, time, tokens, mean ± stddev, delta
8c Analyse — analyzer subagent surfaces: non-discriminating assertions, flaky evals, time/token tradeoffs
8d View — generate_review.py launches eval viewer; add --previous-workspace on iteration 2+
Viewer tabs
Outputs — prompt, rendered output, prev iteration (collapsed), grades (collapsed), feedback textbox
Benchmark — pass rates, timing, token usage, per-eval breakdown, analyst observations
Submit All Reviews → saves feedback.json
Phase 9 · feedback loop · repeats until satisfied
↻
09
Read feedback & improve
iterate
Improvement principles
Generalise — edits must work across many prompts, not just tested examples
Keep lean — remove instructions producing unproductive behavior
Explain why — reasoning beats rigid ALWAYS/NEVER directives
Bundle repeated work — common helper scripts go into scripts/
After improvement
Apply changes to SKILL.md
Rerun all cases into iteration-N+1/ (including baseline)
Launch viewer with --previous-workspace
Loop terminates when: user satisfied, feedback all empty, or no meaningful progress
↖
returns to phase 05 — reruns all test cases into new iteration directory
Phase 10–11 · finalise & ship
10
Description optimisation
optional · recommended
Trigger eval design
8–10 queries that SHOULD trigger — varied phrasings, implicit use cases
8–10 queries that SHOULD NOT — near-misses, adjacent domains, shared keywords
Select best_description by held-out test score (not training score) — avoids overfitting
What makes a good test query
Realistic and messy — "my boss sent me this xlsx file called Q4 sales final FINAL v2.xlsx"
NOT sanitised — "Extract text from PDF" is too simple; Claude handles without skill
11
Package & ship
done
Packaging
package_skill.py <skill-dir> validates first, then zips
Produces skill-name.skill — ready to install
Validation errors block packaging — fix and rerun
Blind comparison (optional)
comparator + analyzer agents judge two versions without knowing which is which
Used when "is the new version actually better?" needs a rigorous answer
Platform variations

Phase 09 loops back to phase 05 until the user is satisfied.
